## 2. Canonical algorithm catalog — real numbers

The canonical source of truth is the plugin catalog under `plugins/<category>/<slug>/`,
loaded at runtime by `apps/backend/algorithm_atlas/plugins/loader.py` into
`PluginRegistry` (`plugins/registry.py`). AtlasCode discovery
(`apps/backend/algorithm_atlas/atlascode/discovery.py`) uses this exact same
loader — never a second hand-maintained algorithm list.

| Metric | Count |
|---|---|
| `manifest.json` files on disk (excluding `plugins/community/`, which is empty) | 250 |
| Successfully load into the registry (**canonical algorithm count**) | **237** |
| Fail to load / collide | 13 |

Breakdown of the 13:
- **8 cellular-automata** (`boids`, `game-of-life`, `seeds`, `wireworld`, and legacy
  copies of `forest-fire`/`langtons-ant`/`rule-110`/`rule-30`) use a pre-schema-v1.0
  manifest shape (`slug` instead of `id`, `complexity` as a bare string, no
  `entry_point`/`version`/`schema_version`). Fixing these would require inventing a
  `time_best`/`time_worst` breakdown that isn't in the source data — left as-is
  rather than fabricating complexity claims. Not relevant to AtlasCode anyway
  (cellular-automata is classified `INTERACTIVE_ONLY`, see §4).
- **3 data-structures** (`b-tree`, `red-black-tree`, `rope`) now pass manifest
  validation (fixed, see §3) but fail at plugin *execution* with
  `Error executing plugin module: 'NoneType' object has no attribute '__dict__'`.
  This is a pre-existing bug in those 3 plugins/loader interaction, out of scope
  for this AtlasCode pass — flagged for a follow-up session, not silently ignored.
- **2 cross-category slug collisions** (`closest-pair` appears under both
  `computational-geometry` and `divide-and-comquer`; `monte-carlo-pi` under both
  `randomized` and `probability`). `PluginRegistry` is a flat `slug -> algorithm`
  dict, so the second registration overwrites the first. This is a genuine
  platform-level ID design tension (intentional cross-listing vs. required
  uniqueness) — documented, not silently patched by renaming, since renaming
  would touch frontend routes/benchmarks outside this session's scope.

### Bug fixed in this session (platform-wide, not AtlasCode-specific)

14 manifests (`distributed-systems/*` ×10, `data-structures/{b-tree,hash-table-probing,
red-black-tree,rope}` ×4) were missing the required `schema_version`/`id`/`version`
fields (2 of the 10 distributed-systems ones were also missing `entry_point` and used
a flat `complexity` string). These are genuine, already-implemented algorithms that
had **never loaded into the running application** — invisible to the Catalog,
benchmarks, and AtlasCode alike, despite `PROGRESS.md` claiming "100% executed."
Fixed by adding the missing schema fields (mechanical transform, no invented
complexity data beyond the single value the file already stated). This alone raised
the working canonical count from 226 to 237.

## 3. Seed script bug (blocking, fixed)

`scripts/seed_atlas_code.py` imported `async_session` from `database.py`, which
does not exist — the actual session factory is `AsyncSessionLocal`. The seed
script had **never successfully run once**. Fixed; verified by actually running
it (see §5).

Two curated problems also had stale `algorithm_slug` values that don't match any
canonical algorithm (`breadth-first-search` → canonical is `bfs`; `minimum-path-sum`
→ canonical is `min-path-sum`), which would have produced dead "Visualize" links.
Fixed to point at the real canonical slugs.

## 5. Family-factory architecture (new)

`apps/backend/algorithm_atlas/atlascode/`:
- `discovery.py` — loads the canonical catalog via the existing `PluginLoader`
- `oracle.py` — runs the **real plugin implementation** (`initialize()` /
  `steps()`, via the SDK's `AlgorithmTestHarness`) to produce ground-truth
  expected output. No hand-typed expected values, ever. Each oracle call also runs
  an independent sanity assertion (e.g. "the output is actually a permutation-sorted
  version of the input") so a broken plugin can't silently produce a wrong answer.
- `families/sorting.py`, `families/searching.py` — generate one problem per
  uncurated algorithm in that category, deterministic fixed-seed test cases
- `coverage.py` — classifies every canonical algorithm (see §4) and writes the
  manifest

`scripts/seed_atlas_code.py` now: discovers the canonical catalog → runs both
family factories → merges with the 19 curated problems → validates (duplicate
slugs, missing visible/hidden test, etc.) → upserts idempotently → prints the
required summary block → writes the coverage manifest. Supports `--dry-run`
(discover/generate/validate, no DB writes) and `--validate-only` (no DB access
at all).

## 7. Known limitations / explicitly deferred (see `docs/atlascode-progress.md`)

- DP, greedy, number-theory, string, divide-and-conquer, cryptography,
  data-structures (92 algorithms, `NEEDS_REVIEW`): good STANDARD_JUDGE candidates,
  no family factory built yet this session.
- Graph, backtracking, tree, computational-geometry (60 algorithms,
  `PROPERTY_JUDGE`): need a property-validator layer (e.g. topological-order
  validity, N-Queens board validity) before they can be judged — not yet built.
- The 3 data-structures plugins with a deeper execution bug, and the 8
  pre-schema cellular-automata manifests — flagged, not fixed (out of scope).
- Frontend: language selector, Run-vs-Submit separation, Atlas AI context
  wiring for AtlasCode — not touched this session; see `docs/atlascode-progress.md`
  for what was already identified before the catalog-coverage correction.
